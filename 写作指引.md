# 无线通信短文写作指引

这份指引默认面向无线通信短文，尤其是 IEEE Wireless Communications Letters 一类的短篇论文。它不是为了替代你对问题本身的理解，而是为了帮助你把一篇短文写得更像一篇短文：结构清楚、章节分工明确、重点集中、读者能顺着你的逻辑走下去。

## 一、先搞清楚短文要解决什么问题

短文不是把长文删短，也不是把几个实验结果简单拼起来。它要服务于一个清楚的目标：回答一个足够具体、足够重要、又能被当前证据支撑的问题。所以在动笔之前，先回答四个问题：

1. 你到底在解决哪个无线通信问题？
2. 这个问题为什么重要，而且为什么是现在重要？
3. 现有方法到底差在哪里，而不是“还不够好”这种空话？
4. 你的方法或发现，到底提供了什么新的 insight、performance gain 或 capability？

如果这四个问题说不清，后面再怎么润色也只是把模糊写得更流畅。

## 二、Introduction 建议采用四段式结构

### 第一段：大背景

第一段讲大背景，但不要写成空泛口号。正确的写法是从一个具体无线场景切入，说明你关注的问题为什么是 fundamental problem，以及在 6G 演进、RIS、超大规模阵列、快速时变环境等新条件下，它为什么变得更难。

这一段的目标不是“显得宏大”，而是把问题的重要性讲具体。比如信道估计为什么关键，不只是因为它重要，而是因为它直接影响后续检测、波束赋形、资源分配或链路可靠性。

### 第二段：小背景与文献综述

第二段是小背景，也就是结构化 literature review。不要把文献一篇一篇平铺罗列，而是要按一条清楚的分类轴来写，例如按 side information、建模方式、估计目标或系统设定来组织。

如果你自己还没有完全理清文献逻辑，可以先把相关论文交给网页版 AI，让它帮你梳理一个分类框架，再由你人工检查是否合理。如果你已经知道这几类文献各自做了什么，也可以直接让 AI 按这个逻辑撰写或润色。

一个常见误区是只看高引用文献。高引用并不一定意味着写法最适合学习。有的文章实验强，但结构一般；有的文章创新点未必最强，但写作的逻辑链非常值得模仿。参考文献的选择，要兼顾内容相关性和写作质量。

### 第三段：总结已有工作并指出缺口

第三段的任务是承上启下。第四段会讲你自己的工作，因此第三段要先把“为什么还需要你的工作”说清楚。这里通常有两种写法。

第一种是总结现有工作的共性局限，比如复杂度高、鲁棒性不足、依赖过强的先验、没有考虑动态环境、训练或标注代价过高。第二种是指出一个 open problem，或者说明现有工作忽略了某个对实际系统很关键的因素。

这一段一定要避免写成“现有工作很多，但仍有待进一步研究”。这种话没有信息量。你必须明确说出，现有方法在你的 operating condition 下为什么不够用。

### 第四段：讲自己的工作与贡献

第四段直接讲自己的工作做了什么。这里不需要把算法细节全写出来，而是要把 contribution 说得干净、并且有层次。通常可以写成 2 到 4 个具体贡献点，或者用一小段连续 prose 说明你的方法、关键机制和主要发现。

要突出 contribution，但不要夸张。短文尤其怕把 novelty 写成 slogan。更好的写法是让读者看到：你提出了什么、为什么这样做、它相对已有方法好在哪里、证据会在后文怎样展开。

## 三、System Model 一般要短、准、够用

system model 往往不是文章的贡献主体，因此篇幅通常控制在半页到一页，不要写成一个独立的大型教程。它的任务是把系统设定、基本假设、符号和目标讲清楚，而且只引入后面真正会用到的变量与记号。

如果你用的是已有模型，比如 MIMO-OFDM、OTFS、RIS-assisted MIMO，这一节主要是做清晰陈述，而不是展示创造力。只有当你的工作本身包含新的建模思路，才值得在这里适当扩展。

一个简单原则是：system model 讲完后，读者应该知道你研究的对象是什么、输入输出是什么、优化或估计目标是什么，以及真正的技术瓶颈落在哪里。如果这些都说清楚了，就不要再继续堆符号。

## 四、符号体系要尽量让读者容易记住

短文篇幅有限，读者通常不会给你太多记忆预算，所以符号体系最好尽量顺着领域里的常见约定来写。一个好的符号体系，不只是“数学上自洽”，还应该尽量减轻读者的阅读和记忆负担。

有一些符号的潜规则，大家会自然而然形成共识。例如，大写 `K` 常用来表示用户数，小写 `k` 表示用户索引；大写 `N` 常用来表示子载波数，大写 `M` 常用来表示天线数。当然，不同方向里 `N` 和 `M` 也可能反过来用，但关键是你一旦选定，就要前后一致，不要在后文里悄悄换轴。

如果收发端都是多天线，比较常见的写法是用 `M_t` 和 `M_r` 分别表示发射端和接收端天线数。这样做的好处是，读者几乎不需要重新学习你的符号体系，就能直接把注意力放在系统关系和方法逻辑上。

矩阵或张量的元素记号，也尽量遵循直觉。比如矩阵 `A` 的第 `i` 行第 `j` 列元素，常见写法就是 `a_{i,j}`。信道矩阵通常写成大写加粗 `H` 或大写加粗 `G`。如果你无特殊原因，尽量不要为了“显得新颖”去发明一套过于个性化的命名。

另一个实用原则是：能用下标的时候，尽量不要再叠加上标；或者反过来，已经用了上标表达某一层含义时，就少用下标再去承载第二层、第三层信息。因为一旦上下标层数太多，读者会在还没理解方法之前，先被符号压垮。

当然，有些复杂系统里，上下标同时出现是不可避免的。比如你既要区分矩阵元素的 `i, j`，又要区分上行和下行，这时可以把元素索引放在下标，把链路方向放在上标括号里，例如 `H_{i,j}^{\mathrm{UL}}`。这里的 `UL` 不要用斜体，而要用正体，因为它代表的是 `uplink` 这个单词，而不是一个待优化的数学变量。对应地下行也可以写成 `\mathrm{DL}`。

总之，符号体系的目标不是炫技，而是降低读者的理解成本。你让读者越容易记住你的符号，他就越容易跟上你的推导、系统模型和算法结构。

## 五、第三章和第四章通常是全文核心

### 第三章：方法论

这一章是核心方法。比较稳妥的组织方式是用多个 subsection，把逻辑拆开来写，而不是一大段把所有内容挤在一起。

一个常见结构如下：

1. 第一个 subsection 写 problem formulation，或者把它单独成章。
2. 第二个 subsection 写主体方法，可以按模块写，也可以按关键步骤写。
3. 第三个 subsection 总结 overall algorithm，给出伪代码，并说明运行流程、收敛准则、复杂度或实现代价。

如果你的方法本质上是一个 pipeline，就按照 input -> processing -> output 的顺序写。如果你的方法是多模块协同，就按模块之间的数据流或依赖关系写。无论哪种写法，都要先讲 intuition，再讲 detail。不要一上来就把公式和伪代码砸给读者。

### 第四章：仿真与讨论

第四章不是简单报数，而是要证明第三章提出的方法为什么值得写。你可以做两类事情。

第一类是横向对比：和已有 baseline 比，展示性能增益、复杂度收益或鲁棒性优势。

第二类是纵向理解：如果这个方向本身比较新，缺少标准 baseline，那就对自己的方案做更细的纵向比较，例如不同模块、不同参数、不同 operating condition 下的表现，从而提供新的理解。

无论哪种，正文都要解释趋势背后的原因，而不是只说“图 2 表明我们的方法最好”。真正有说服力的写法，是说明在哪些条件下增益最明显、为什么会这样、这对应怎样的系统含义。

## 六、短文里的图通常控制在 1 到 2 张

短文篇幅有限，图一般给 1 到 2 张就够了。如果内容丰富，可以在一张图中放 2 到 3 个子图，左右或上下排列，但前提是每个子图都服务于主线，而不是为了“显得做了很多实验”。

图正文和 caption 要分工明确：

1. caption 负责系统设置、参数、场景说明。
2. 正文负责比较对象、主要趋势和工程含义。

如果 caption 已经写了阵列规模、频率、scatterer 数量等配置，正文就不要再机械重复。正文应该回答的是：这张图说明了什么，为什么会出现这个趋势，以及这个趋势为什么重要。

## 七、怎样借助 AI，而不是被 AI 带着跑

### 场景一：从零到一

先找 5 篇左右参考论文，优先挑“写法值得学”的，而不是只挑“引用最高”的。然后把这些论文交给 AI，让它提炼：

1. Introduction 是怎样铺垫的。
2. 文献综述按什么逻辑分类。
3. contribution 是如何展开的。
4. 图和正文如何呼应。
5. 哪些句式或段落组织方式值得复用。

这个阶段建议用 plan 模式。让 AI 先说它准备怎样更新知识库、保留哪些共性规则、哪些规则只是这几篇论文的局部写法。这样可以避免模型把原来积累下来的关键规则误删。

### 场景二：局部修改

如果你已经有草稿，可以让 AI 专门处理局部问题，例如：

1. 重写 Introduction 第二段，让 literature review 更有结构。
2. 压缩 system model 到半页，同时保留必要符号。
3. 把 method section 拆成更清楚的三个 subsection。
4. 给 results 增加解释句，而不是只报数字。

局部修改时，给 AI 的输入要尽量具体，至少包括：当前段落、你想保留的核心意思、参考 baseline 或文献、目标 venue，以及你最担心的问题。

### 场景三：外部评审

AI 有一个问题：它容易在当前对话里自洽，也容易迎合作者。一个问题如果在最开始没被提出，后面可能就一直被忽略。所以很有必要在另一个窗口、新开一个对话，甚至换一个 agent，让它站在审稿人的角度重新评判你的文章。

旁观者清。你可以把它当作一个廉价、快速、但不应盲信的外部评审机制。

## 八、几个很常见的坑

1. 前后命名不一致。前面叫 dynamic prior，后面变成 map prior，再后面又变成 side information，读者会很累。
2. 只会说“现有方法存在局限”，但说不出具体局限是什么。
3. system model 太长，抢了方法部分的篇幅。
4. method 章节没有先讲 intuition，直接堆公式。
5. results 章节只有“图表明”，没有“为什么”。
6. `for`、`as` 等介词和连接词乱用，导致技术关系表达不准确。
7. 模型偷懒时不拆任务，最后得到一个结构空洞但表面流畅的成稿。

## 九、最后的判断标准

核心标准不是“像不像论文”，而是“有没有把问题、方法和证据讲清楚”。如果没有 AI，你理论上也应该能高质量完成这件事，只是速度会慢很多。AI 的价值在于提速、补盲区、帮你保持结构纪律，而不是替你形成判断。

随着模型能力快速增强，我们手里确实像是多了一种很强的工具。但工具越强，对使用者的理解力、审美和判断力要求就越高。把重复劳动交给 AI 之后，更应该把时间花在真正决定文章质量的地方：问题意识、逻辑结构、证据强度，以及你对自己文字好坏的辨别能力上。
